Why this exists

An agent that re-learns your project every session is slow and error-prone. These skills install and maintain a docs/ knowledge base so the agent (or a teammate) understands a feature by reading one page instead of scanning the code — and never reopens settled decisions or repeats a mistake already paid for.

It's a mix of Matt Pocock's skills (MIT) and Pedro Mota's optimized additions, focused on project work and planning.

The knowledge base

Six artifacts under docs/ (plus CONTEXT.md at the root), each answering a different question. /setup-pedro-mota creates them all and writes a CLAUDE.md that explains them.

ArtifactQuestionTense / role
CONTEXT.mdwhat the words meanglossary (vocabulary)
docs/adr/why we decided this waydecisions (the why)
docs/system/what the code does todaypresent · source of truth
docs/plans/what we're going to dofuture · ephemeral
docs/pending/what's left openloose ends
docs/learnings/where we already erredlessons
Golden rule: read docs/system/ before coding, update it after (with /sync-doc). The doc must never diverge from the code — if it does, the doc is wrong: fix it.

The work loop

/grill-with-docsdecide · pin CONTEXT.md + ADRs
/to-planwrite the plan in docs/plans
/handoffpass the plan to a NEW agent — execute or continue
implementthe new agent builds it (optionally /tdd)
/sync-docupdate docs/system
/to-plan donearchive the plan

/handoff is the bridge between planning and building: once the plan is written, it produces a ready-to-paste prompt (saved under .handoff/ and opened) that a new agent/chat uses to pick up — to execute the plan or to continue planning — wired to all the docs. Along the way: /to-pending for anything deferred, and a note in docs/learnings/ if you got bitten.

Board-driven alternative. Instead of (or alongside) /handoff, run /to-tasks to publish the plan as Click Notes tasks + subtasks — each slice gated in its title ([READY FOR DEV] to code, [PLANNING] to leave alone, [WIP] while worked). Autonomous agents then run /do-task to pull the next ready task, read its plan + ADRs, build, test, and mark it done. Publish once, let agents pull.

Grilling & planning
Matt Pocock

/grill-with-docs

Interviews you relentlessly to stress-test a plan against the existing domain model — sharpening terminology in CONTEXT.md and writing ADRs inline as decisions crystallize.

When: before committing to a design, to find the holes.

Matt Pocock

/grill-me

The lighter grilling: walks each branch of the decision tree one question at a time until you reach shared understanding. No docs awareness.

When: quick stress-test of an idea or plan.

Pedro · new

/to-plan

Distills the decisions from a grilling into an implementation plan under docs/plans/ (goal, closed decisions, changes per file, tests, acceptance criteria). Cites & promotes ADRs. /to-plan done <slug> archives it to done/ with a ✅ badge.

When: right after a grilling; and when the feature ships.

/to-plan → writes docs/plans/2026-06-14-x.md
Pedro · new

/to-pending

Records a loose end ("we'll look at it later") as a detailed, actionable item in docs/pending/ — why deferred, impact, next step. done mode resolves it.

When: you defer an edge case, a decision, or tech debt.

Matt Pocock

/ubiquitous-language

Extracts a DDD-style glossary from the conversation, flagging ambiguities and proposing canonical terms → UBIQUITOUS_LANGUAGE.md (sibling to CONTEXT.md).

When: nailing down domain vocabulary.

Docs & handoff
Pedro · new

/sync-doc

At the end of a feature, syncs docs/system/feature-*.md with the real code — creates it if missing, fixes every divergence claim-by-claim, and updates the "Topic map" index. Every cited name must pass git grep.

When: after finishing any feature/fix.

Pedro · adapted

/handoff

Passes the baton to another chat: writes a handoff doc + a ready-to-paste prompt under .handoff/ and opens it. The prompt is tailored — continue the planning or execute the plan — and points at the knowledge base in reading order.

When: ending a session; continuing elsewhere.

Tasks & autonomous execution · Click Notes board
Pedro · new

/to-tasks

Publishes a plan from docs/plans/ as Click Notes tasks + subtasks — the plan becomes one root task, each vertical slice a subtask, gated in the title ([READY FOR DEV] / [PLANNING] / [WIP]). Each task carries pointers to the plan + ADRs so an agent can build it without reopening decisions.

When: after /to-plan, to hand work to autonomous agents.

plan → root task + slice subtasks, gated for pickup
Pedro · new

/do-task

An autonomous agent grabs the next [READY FOR DEV] task, reads its linked plan + ADRs + feature docs, claims it ([WIP]), implements + tests, and marks it done. Never codes a [PLANNING] task; refuses to code with no detailed plan.

When: draining the board — one slice per run.

Click Notes — meetings & memory
Pedro · new

/clicknotes-meeting

Reads a meeting's transcript + thin notes → rewrites/complements the notes faithfully, and turns action items into tasks linked to the meeting — deduping against tasks the org already has instead of creating twins.

When: wrapping up a meeting.

Pedro · new

/clicknotes-memory

Creates/updates a block in the company knowledge graph, obeying its living conventions (mandatory header, stable ID, fixed sections, reused tags) and wiring it into the graph (wikilinks, parent/related, hub index).

When: saving a durable fact/decision to memory.

Pedro · new

/clicknotes-recall

Searches the knowledge graph for the current topic and walks the connections (wikilinks, backlinks, tag context) to ground the conversation in what the org already knows. Read-only.

When: before answering a domain question.

Pedro · new

/clicknotes-tasks

Reconciles the task board against reality: reviews open tasks, marks finished ones done and partial ones in-progress with what remains. Evidence-based — never closes a task it can't prove is done.

When: keeping the board honest.

Engineering

When: hard bugs or performance regressions.

Matt Pocock

When: you're lost in an unfamiliar area.

Matt Pocock

When: sanity-check a data model or explore UI options.

Install

Each skill is a folder with a SKILL.md. Clone the repo and symlink the skills into your agent's skills directory (Claude Code: ~/.claude/skills):

git clone https://github.com/pvieira-design/pedro-mota-skills.git
cd pedro-mota-skills
# symlink every skill (top-level and nested, e.g. skills/clicknotes/*)
find skills -name SKILL.md | while read -r f; do
  d=$(dirname "$f"); ln -sfn "$PWD/$d" ~/.claude/skills/"$(basename "$d")"
done

Then invoke any skill as /<skill-name>. On a new repo, start with /setup-pedro-mota.
